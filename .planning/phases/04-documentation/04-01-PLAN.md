3. **## Using prompts in Flycut** (DOCS-01) -- Short section: The sync URL is pre-configured in Flycut. Users can also paste the URL into Settings > Prompts > JSON URL > Sync Now.

4. **## Adding a prompt** (DOCS-02) -- Numbered step-by-step:
   1. Choose the right category directory (`prompts/coding/`, `prompts/writing/`, `prompts/analysis/`, `prompts/creative/`)
   2. Create `prompts/<category>/<kebab-case-id>.md` (filename becomes the prompt ID; lowercase a-z, 0-9, hyphens only; must be unique across all categories)
   3. Write the required YAML frontmatter block (show minimal example with title + version: 1)
   4. Write the prompt body below the frontmatter
   5. Run `bash scripts/build.sh` locally to verify (show expected output: "Built prompts.json: 23 prompt(s), catalog version 2 / Validation passed...")
   6. Submit a pull request

   Include a **complete example prompt file** block using the summarize-text example from the research doc.

   Include the **frontmatter field reference table** with all 6 fields (title, version, category, description, tags, variables), their types, required status, and descriptions. Note which fields are sent to Flycut (title, version, category, content via body) vs metadata-only (description, tags, variables).

5. **## Updating a prompt** (DOCS-03) -- Explain: edit the content AND bump the version field. Include the version bump warning as a prominent blockquote:
   > **Important:** Every content change to a prompt file **must** include a version bump (e.g., `version: 1` to `version: 2`). Without this, Flycut's sync sees `remote.version == local.version` and skips the update silently. Users will not receive the change.

6. **## Template variables** (DOCS-04) -- Explain:
   - `{{clipboard}}` is built-in: contains clipboard text at paste time
   - Other `{{variable}}` placeholders are looked up from user-defined key-value pairs in Flycut Settings > Prompts > Template Variables
   - Unknown variables are left as literal text in the output
   - Show a short example prompt body demonstrating `{{clipboard}}`

7. **## Versioning rules** (DOCS-06) -- Document all five rules from the design spec (PROMPTS-REPO-DESIGN.md section "Versioning Rules"):
   1. Adding a new prompt: set version: 1
   2. Updating prompt content: bump the prompt's version (e.g., 1 to 2)
   3. Fixing a typo before anyone synced: can keep version: 1 if certain; otherwise bump
   4. Never decrease a version number (Flycut only updates when remote > local)
   5. Catalog version (catalog.yaml): bumped by maintainers on merge; contributors do not need to bump this in PRs
   Note: rule 5 follows the research recommendation that maintainers handle catalog version bumps to avoid merge conflicts.

8. **## Building locally** -- Show the build command (`bash scripts/build.sh`) and expected output. Note the prerequisite: `jq` must be installed (`brew install jq`).

9. **## Prompt catalog** (DOCS-05) -- Table of all 23 prompts grouped by category. Use the exact data from 04-RESEARCH.md "Complete Prompt Catalog Data" section. Format:
   - Subheading per category: ### Coding (8 prompts), ### Writing (6 prompts), ### Analysis (5 prompts), ### Creative (4 prompts)
   - Table columns: Prompt ID, Title, Description
   - Sorted alphabetically within each category (matching prompts.json sort order)

10. **One-time setup note** (at the bottom or in a "Maintainer notes" section) -- After the first successful CI run, enable GitHub Pages in Settings > Pages > Source: Deploy from branch > gh-pages > / (root). This addresses the STATE.md blocker.
